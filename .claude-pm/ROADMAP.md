# Roadmap

The roadmap describes outcomes, not dates. Dates belong on work items only when externally meaningful.

## Phase 0 — Repository reconciliation

Outcome: the starter framework is safely integrated with the intended repository.

Exit conditions:

- Authoritative documents and application commands are identified.
- Existing local changes are preserved.
- Framework paths and statements match repository reality.
- Dashboard and application checks complete or failures are documented.

## Phase 1 — Baseline

Outcome: the tracker accurately represents verified current work.

Exit conditions:

- Initial work items are sourced from repository evidence.
- Dependencies, priorities, owners, and acceptance criteria are useful.
- Unknowns are explicit and assigned for resolution.

## Phase 2 — Delivery

Outcome: normal development uses the tracker as a lightweight coordination aid.

Exit conditions:

- Active work is linked to work-item IDs.
- Completed work includes validation evidence.
- Status and roadmap remain concise and current.

## Phase 3 — Maintenance

Outcome: the framework stays useful without creating process overhead.

Exit conditions:

- Stale items are reviewed or archived.
- Locked constraints and change history remain auditable.
- The dashboard can be regenerated from the canonical backlog at any time.
